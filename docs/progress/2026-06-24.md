## ⚠️ DECISIÓN ABIERTA — estrategia de identidad/cuentas (GitHub + SaaS)
Surgió porque usar cuentas separadas por proyecto (`empresas-ia-dev`, `criza-dev`) generó fricción (push falla si la cuenta activa de `gh` no es la correcta; login de Vercel/Railway atado a una cuenta de GitHub que se quiere retirar). **NADA decidido ni ejecutado todavía** — Sebas pidió no apurar y validar.

**Roca firme (estándar GitHub, a validar con docs oficiales):**
1. Una persona = una sola cuenta personal de GitHub.
2. La empresa = una **organización** (contenedor de repos, transferible).
3. Las personas acceden como **miembros** de la org con su cuenta personal.
4. No hace falta una cuenta-usuario de GitHub por proyecto.

**Criterio (trade-off, sin decidir):** ¿cuentas SaaS (Vercel/Railway/Neon) bajo un **email de empresa** (mejor para vender) o bajo la personal (más simple)?

**Patrón propuesto (vendible):** identidad de empresa = **email corporativo** (root de SaaS/billing) + **org de GitHub** (repos); personas commitean con su cuenta personal como miembros; credenciales en un vault. Venta = transferir la org + entregar email/vault.

**Paso de bajo riesgo (no ejecutado):** agregar `sebasbizzi` como Owner de la org `empresas-ia` para que el `git push` deje de fallar (hoy requiere `gh auth switch --user empresas-ia-dev`).

**Preguntas pendientes para decidir:** ¿vender en serio/pronto? ¿socios/inversores? ¿cuántas empresas en paralelo? · Validar la "roca firme" contra docs oficiales de GitHub/Vercel/Railway. · Si se confirma → escribir la sección "Identidad y acceso" en `docs/playbook.md` (Capa 0).
